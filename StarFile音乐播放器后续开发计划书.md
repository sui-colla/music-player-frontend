# StarFile 音乐播放器后续开发计划书

## 1. 当前版本概况

当前项目已完成 v0.1 版本，具备音乐播放器的基础闭环能力：

- 本地静态服务与一键启动脚本。
- 首页、音乐库、喜欢、最近播放、播放队列、歌单等主要视图。
- 播放、暂停、上一首、下一首、进度条、音量、静音等播放控制。
- 顺序播放、单曲循环、随机播放三种播放模式。
- 播放队列、收藏、最近播放、歌单、主题、音量等本地保存。
- IndexedDB 本地歌曲导入、LRC 歌词解析与同步歌词显示。
- 基础响应式布局、提示状态、错误状态和 Windows 打包目录雏形。

v0.1 已经不是单纯静态页面，而是一个可运行的本地音乐播放器原型。后续重点应从“继续堆页面”转为“提升播放器可信度、导入体验和展示完成度”。

## 2. 后续开发目标

后续版本的核心目标是把 StarFile 从功能原型推进到可展示、可长期维护、可打包分发的完整作品。

主要目标：

- 打开应用后即有完整可体验内容。
- 本地音乐导入流程更顺手、更可靠。
- 播放、队列、歌词、主题等状态保持一致。
- 视觉风格稳定，不依赖临时覆盖样式。
- 桌面端打包体验逐步完善。

## 3. 版本路线

### 3.1 v0.1.1 稳定修复版

优先级：最高

目标：修复 v0.1 中影响第一印象和基础可信度的问题。

计划内容：

- 增加 3-5 首默认可播放样例歌曲。
- 为默认歌曲补充封面、时长、歌词和基础元数据。
- 修复播放队列移除入口缺失的问题。
- 读取并应用已保存的主题设置。
- 统一左侧导航文案和实际视图含义。
- 为本地歌曲删除增加二次确认，避免误删。
- 优化空状态：无歌曲、无歌词、无搜索结果、无队列时给出更明确反馈。
- 检查小屏幕布局，确保底部播放器、右侧队列和首页内容不重叠。

交付标准：

- 首次打开应用不导入歌曲也能完整播放样例音乐。
- 刷新页面后主题、音量、播放模式、收藏等状态正确保留。
- 队列歌曲可以正常添加、播放、移除。
- 导航名称和进入页面的内容一致。

### 3.2 v0.2 本地音乐增强版

优先级：高

目标：把“本地音乐播放器”做成项目的核心卖点。

计划内容：

- 支持拖拽导入音频文件和 LRC 歌词文件。
- 支持批量导入前编辑歌曲标题、歌手、专辑和歌词。
- 显示导入进度、成功数量和失败数量。
- 支持为已导入歌曲重新绑定或替换歌词。
- 支持编辑已导入歌曲的基础信息。
- 删除本地歌曲时同步清理队列、收藏、歌单和最近播放记录。
- 增加浏览器存储空间不足或 IndexedDB 不可用时的错误提示。
- 增加本地音乐专属视图，集中展示用户导入的歌曲。

交付标准：

- 用户可以通过点击或拖拽完成本地歌曲导入。
- 导入失败时能明确知道失败原因。
- 已导入歌曲可继续编辑信息和歌词。
- 本地音乐视图可以独立搜索、播放和管理。

### 3.3 v0.3 沉浸播放与展示版

优先级：中

目标：提升作品展示效果，让应用更像完整音乐软件。

计划内容：

- 新增沉浸播放页或大歌词模式。
- 展示大封面、当前歌词、下一句歌词和播放进度。
- 根据封面或主题生成播放背景色。
- 增加轻量音频可视化效果，例如频谱条或波形动画。
- 优化歌词滚动动画，减少频繁重排。
- 播放栏增加更清晰的播放模式、喜欢状态和队列入口。
- 使用更统一的图标体系替换零散字符图标。

交付标准：

- 播放音乐时有完整的沉浸展示界面。
- 歌词、封面、播放状态之间同步自然。
- 视觉效果提升但不影响基础播放性能。

### 3.4 v0.4 桌面端完善版

优先级：中

目标：完善 Windows 桌面端使用体验。

计划内容：

- 将窗口最小化、最大化、关闭按钮接入桌面宿主能力。
- 完善打包脚本和安装包流程。
- 设置应用图标、窗口标题和默认窗口尺寸。
- 检查 WebView2 Runtime 依赖提示。
- 记录桌面端启动失败、端口占用、静态资源缺失等异常情况。

交付标准：

- 用户可以通过安装包安装并启动播放器。
- 窗口控制按钮在桌面端真实可用。
- 打包流程有明确脚本和说明。

## 4. 技术整理计划

当前代码以原生 HTML、CSS、JavaScript 为主，适合快速迭代。后续如果功能继续扩大，需要逐步整理代码边界。

短期整理：

- 拆分 `app.js` 中的数据存储、播放控制、渲染逻辑和事件绑定。
- 将时间格式化、歌词解析、IndexedDB 操作提取为独立工具模块。
- 清理重复 CSS 和临时覆盖样式。
- 将样式按布局、侧栏、首页、歌曲列表、队列、播放器、弹窗、响应式拆分。

中期升级：

- 若继续扩展为长期项目，可迁移到 Vue 3 或 React。
- 引入组件化结构，拆出 Sidebar、PlayerBar、SongList、QueuePanel、LyricPanel、ImportDialog 等组件。
- 用统一状态管理维护播放状态、音乐库、歌单和用户偏好。

## 5. 测试重点

每个版本发布前需要重点测试以下场景：

- 首次打开应用是否能看到并播放默认歌曲。
- 本地歌曲导入、刷新、再次播放是否正常。
- LRC 歌词是否能正确解析和同步高亮。
- 播放、暂停、上一首、下一首、拖动进度条是否稳定。
- 顺序播放、单曲循环、随机播放是否符合预期。
- 收藏、歌单、队列、最近播放刷新后是否保留。
- 删除歌曲后相关引用是否被清理。
- 主题切换、音量设置、播放模式是否持久化。
- 小屏幕和桌面宽屏下是否有文字溢出或布局重叠。
- 打包后的桌面端能否启动并加载本地资源。

## 6. 风险与应对

### 6.1 音频资源版权风险

风险：默认示例歌曲如果使用真实商业音乐，可能存在版权问题。

应对：使用开源音乐、无版权音频、自制测试音频或仅保留用户本地导入能力。

### 6.2 浏览器存储限制

风险：IndexedDB 存储大体积音频文件时可能触发空间限制。

应对：导入失败时给出明确提示，并在后续版本增加存储占用展示。

### 6.3 单文件逻辑膨胀

风险：`app.js` 功能继续增加后维护成本快速上升。

应对：v0.2 前优先拆分存储、播放、渲染和工具函数。

### 6.4 样式覆盖过多

风险：CSS 后续覆盖会导致响应式和主题切换越来越难调。

应对：尽快做样式分层整理，减少底部集中覆盖规则。

## 7. 推荐执行顺序

建议按以下顺序推进：

1. 完成 v0.1.1：默认样例歌曲、队列移除、主题读取、导航一致性。
2. 完成 v0.2：拖拽导入、批量编辑、本地音乐管理、歌词重绑。
3. 整理代码结构：拆分工具模块和样式模块。
4. 完成 v0.3：沉浸播放页、歌词展示增强、音频可视化。
5. 完成 v0.4：桌面端窗口控制、安装包和打包说明。

这个路线可以保证每个阶段都有明确成果，也能避免过早引入复杂后端或框架迁移导致项目节奏失控。
